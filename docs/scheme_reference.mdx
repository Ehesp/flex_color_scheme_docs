# Scheme Reference

This table lists all current built-in color schemes in FlexColorScheme, the version it was released with, its enum value, short name and description. It shows a small thumbnail of each color scheme in light and dark theme mode. Clicking on a thumbnail opens a high-resolution version of the image.

The column **Swap on M3** shows if a scheme will when the `FlexScheme.light/dark` and `FlexThemeData.light/dark` properties `swapLegacyOnMaterial3` and `useMaterial3` are set to true, swap its `secondary`, `tertiary` colors and their container colors. This is only set to true, if doing so makes a scheme originally designed for the Material 2 color system, a better fit for intended color design when using the Material 3 color system. If it does not improve the fit or if it is already a good fit, it is not set to swap. The indicated swap only happens if you also set both `swapLegacyOnMaterial3` and `useMaterial3` to `true`.

Color schemes released starting from version 5.0, were designed with usage in the Material 3 color system in mind. They can even be native Material 3 schemes, based on pre-seeded colors or slight variations of them. Color schemes used as examples in the Material 3 design guide, most prominently the one often referred to as M3 baseline, are also included.

FlexColorScheme can use the secondary color of a built-in scheme as a key color to seed generate secondary colors. The secondary color and its container, are in the Material 3 color system used for less prominent components in the UI. It is in the design typically a less chromatic version of the primary color's hue, or a hue close to it. If the hue of a scheme's secondary color, or your custom scheme, differs a lot in hue from the primary color, then it may be preferred to not use the secondary color as a key color when seed generating a color scheme. This will then in such cases produce a scheme with better M3 design fit, but you will of course not see the hue of the secondary color in the scheme in the actual color scheme. The Flutter SDK's more limited seed generation offered by `ColorScheme.fromSeed`, can only use one seed color, with it you have no choice.

The column **Second Key** indicate if it is preferred to exclude the scheme's secondary color when making a seed generated Material 3 color scheme from it, in order to get a scheme that fits the secondary color's design intent in the Material 3 color system design. The recommendations are based on that `swapLegacyOnMaterial3` is also set to `true`. The recommendations are:

 - **exclude**, prefer not using secondary key.
 - **borderline**, the resulting color is a bit far of from M3 intent, but not too bad if you flex a bit.
 - **may use**, it is not exactly the same, but close enough to not matter.
 - **can use**, this works fine with the secondary key being used, will be almost same even if not used.

The tertiary colors don't have such design restrictions, so there using it as key color or relaying on the default Material 3 color systems +60 degree hue rotation, is a matter of design choice, either will work fine, just gives different results.
